  1. Anatomy of a Complete Range Conversation

Figure A-1 illustrates the process flow when measuring a single range from the point of view of (user) Host1 connected to a P4xx operating as a Node. In this figure, P4xx-1 is the Requestor and P4xx-2 is the responder.

Fig. A-1: Broadspec UWB antenna with radiating point indicated

  1. REQUEST: HOST1 issues a SendRangeRequest message to P4xx-1 , which is connected via Ethernet, USB, or Serial.
  2. REQUEST PACKET: P4xx-1 emits a UWB packet in range request form targeted at P4xx-2.
  3. CONFIRM: P4xx-1 responds with a SEND_RANGE_CONFIRM message.
  4. RESPONSE PACKET: P4xx-2, if targeted by HOST1 as the RESPONDER, immediately responds with a UWB response packet.
  5. DATA INFO: Any user data in the request packet is reported to HOST2 (if connected).
  6. SCAN INFO: If configured to send scan info, P4xx-2 will send SCAN_INFO data to HOST2.
  7. SCAN INFO: Upon reception of the response from P4xx-2 and if configured to send scan info, P4xx-1 will report scan data to HOST1.
  8. DATA INFO: Any RESPONSE_DATA in the response packet will be reported to HOST1.
  9. RANGE INFO: Finally, P4xx-1 will compute distance measurements and report a RCM_RANGE_INFO message to HOST1.

NOTE 1: DataInfo is reported to ANY Host connected to ANY P4xx that “overhears” a packet (promiscuous operation.)

NOTE 2: ScanInfo is only sent to the Host if the Host has pre-configured the P4xx to send it.

NOTE 3: A single MESSAGE_ID in HOST1’s originating REQUEST message will be echoed at all end-points of this information flow. All RANGE, DATA, and SCAN INFO messages, at source, sink, and promiscuous Hosts, can be tied together using this single MESSAGE_ID. Thus MESSAGE_ID becomes an important tool when logging data and post-processing for full system connectivity and range analysis. In fact, the Host should rely on MESSAGE_ID for correlating messages, rather than any particular order or timing of RANGE, DATA, and SCAN INFO messages.
